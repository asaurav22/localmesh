## Context
The Control Plane registry needs to store service locations — the host, port, and metadata of every registered microservice. A storage mechanism must be chosen to back this registry.

## Decision
Use a plain Python dictionary as an in-memory store, protected by a `threading.Lock()` for concurrent access safety.

## Alternatives Considered

**SQLite**
A lightweight file-based relational database. Would survive Control Plane restarts but adds schema management, query overhead, and file I/O complexity with no meaningful benefit in a local development context.

**Redis**
A production-grade in-memory store with persistence, pub/sub, and TTL support built in. Powerful but introduces an external process dependency just to run local development tooling. Overkill for this scope.

## Reasoning
In LocalMesh, every service re-registers itself with the Control Plane on startup. This means the registry self-populates automatically every time the mesh starts — persistence across restarts adds complexity with zero practical benefit locally. The in-memory dict is fast, zero-dependency, and sufficient for the problem scope.

## Trade-offs Accepted
- Registry data is lost if the Control Plane process restarts
- No built-in persistence or replication
- Not suitable for multi-node or distributed deployments

## Production Equivalent
In production, this registry would be backed by **etcd** (used by Kubernetes) or **Consul** (used by many service mesh implementations). Both provide distributed consensus via the Raft algorithm, survive node failures, and support watch-based push notifications — solving problems that simply don't exist in a local single-machine environment.
