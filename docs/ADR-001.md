# ADR-001: FastAPI (Python) vs Go for Sidecar Runtime

## Context
The sidecar proxy is the most performance-sensitive component in LocalMesh. Every single inter-service request flows through it. The choice of runtime directly impacts latency, throughput, and concurrency handling. A decision was needed on which language and framework to use.

## Decision
Use FastAPI (Python) with asyncio for the sidecar proxy runtime.

## Alternatives Considered

**Go (net/http or Gin)**
Go is the industry standard for writing proxies and networking tools. It compiles to a single binary, has true OS-level concurrency via goroutines, and adds sub-millisecond overhead per request. Envoy aside, most production proxies — Traefik, Caddy, and Linkerd's proxy — are written in Go for exactly these reasons.

**Node.js (http-proxy)**
Event-driven and non-blocking by default. Reasonable for a proxy workload but lacks the type safety and structured async model that makes concurrent code maintainable. Not commonly used for infrastructure tooling in production.

## Reasoning
LocalMesh is a learning project. The primary goal is understanding distributed systems concepts — not building a production-grade proxy. FastAPI with asyncio provides genuine asynchronous request handling via Python's event loop, which is sufficient for a local development mesh handling tens of requests per second. Using Python reduces the barrier to reading, modifying, and understanding the code — which is the entire point of this project.

## Trade-offs Accepted
- Approximately 2-5ms higher proxy latency compared to a Go implementation
- Python's GIL (Global Interpreter Lock) prevents true CPU parallelism across threads
- Higher memory footprint per instance compared to a compiled Go binary
- Not suitable for production proxy workloads at any meaningful scale

## Production Equivalent
In production, this sidecar would be replaced by **Envoy** — written in C++ and capable of handling millions of requests per second with sub-millisecond overhead. Envoy is the data plane used by Istio, AWS App Mesh, and most major service mesh implementations. The xDS protocol it speaks would replace our custom Control Plane HTTP API entirely.
